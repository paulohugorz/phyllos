PHYLLOS · Fashion OS · Fase 2

Plano Mestre — Interface Web

Owner: Founder / CEO  ·  Última revisão: 10 jun 2026

A PHYLLOS tem um Fashion OS v1 funcional operando via terminal com FastAPI + SQLite + 22 agentes Claude. O gargalo hoje é operacional: só funciona para quem domina linha de comando. A Fase 2 converte essa infraestrutura em interface web usável por qualquer membro do time. A stack foi decidida (Next.js 14 + Tailwind + shadcn/ui), o custo de infra é R$0/mês, e o escopo está travado em cinco funcionalidades reais: coleções, peças, ficha técnica editável, galeria de imagens e exportação de PDF.

O critério de encerramento da Fase 2 é verificável: qualquer membro do time cria coleção + peça + exporta PDF sem abrir terminal em menos de 15 minutos.

Mapa de Riscos

R1 — PDF inferior ao template Markdown atual

Prob. Alta Impacto Alto

O PDF gerado pelo @react-pdf/renderer pode ter layout inferior ao template Markdown que o time já usa. Se isso acontecer, o time ignora a plataforma e volta ao terminal.

Mitigação

  • D3 (ficha de referência anotada) é pré-requisito obrigatório para o Sprint 5.
  • Testar o PDF com o fornecedor real antes de declarar Sprint 5 concluído.
  • Se @react-pdf/renderer não entregar, avaliar pdfmake ou jsPDF como alternativa — não depois.
  • Critério: founder responde “sim, enviaria para fornecedor” antes do sprint ser fechado.

R2 — Scope creep técnico durante os sprints

Prob. Alta Impacto Médio

Cada sprint vai gerar ideias de feature nova. O agente de código não tem autonomia para rejeitar; o founder pode aprovar sem perceber que está postergando o critério de encerramento.

Mitigação

  • Qualquer feature não listada neste plano vai para docs/planejamento/fase3-backlog.md sem discussão.
  • Qualquer código que chame claude ou anthropic dentro de frontend/ é bloqueado nesta fase.

R3 — Nenhum usuário real testando durante o build

Prob. Média Impacto Alto

A interface pode ser construída “corretamente” segundo os wireframes e ser inutilizável na prática. Sem teste real durante o desenvolvimento, o problema aparece só no Sprint 6.

Mitigação

  • Ao final dos Sprints 2 e 3 (não só no 6), o founder executa o fluxo principal por 10 minutos.
  • Feedback capturado em docs/feedback/sprint-[n]-fundadora.md com no máximo 3 itens de ajuste.

R4 — Dívida técnica de SQLite bloqueando a Fase 4

Prob. Baixa Impacto Alto

Sem Alembic configurado agora, qualquer alteração futura no schema exige migração manual com risco de corrupção de dados.

Mitigação

  • Alembic é obrigatório no Sprint 0. Verificação: alembic current deve retornar head.
  • Toda alteração de schema usa alembic revision --autogenerate — documentar no README agora.

R5 — Perda de dados do SQLite (arquivo local)

Prob. Baixa Impacto Alto

O banco SQLite está no disco local do founder. Um acidente (formatação, falha de HD) zera o Fashion OS inteiro.

Mitigação

  • CFO recomenda: Litestream para replicação contínua do SQLite para Cloudflare R2 (~R$5/mês).
  • Alternativa gratuita: script de backup diário via cron para Google Drive ou Time Machine.
  • Implementar no Sprint 0: scripts/backup-db.sh com cron entry documentado.
